Runtime MDX in a Parachute vault app

A note whose extension is mdx is compiled in the browser at read time — markdown, live components, and Mermaid diagrams, all from the same vault bytes. No build step, no server, no MDX pipeline. Edit the note; the behavior changes on next open.

01 The one idea

The vault already serves every note over the same REST API. So the app treats an mdx note like any other fetch — then, instead of rendering it as static markdown, it compiles the source to a React component on the client and mounts it against a fixed registry of components. The note is still just text in the vault; the app is the lens that makes it run.

the whole path — fetch → compile → registry, with a markdown fallback

02 The core

The entire renderer is @mdx-js/mdx's evaluate against the React JSX runtime, plus a registry and an error boundary. Roughly:

MdxNote.tsx (abridged)

// compile the fetched source to a component, in the browser
const mod = await evaluate(source, { ...jsxRuntime, baseUrl })

// the vocabulary a note may draw on — anything else degrades to text
const COMPONENTS = { Term, Checklist, LayerQuiz, ObjectionSim,
                    Mermaid, pre: MermaidFence }

// render; a runtime error falls back to plain markdown so a
// broken note still reads
<MdxErrorBoundary source={source}>
  <mod.default components={COMPONENTS} />
</MdxErrorBoundary>

Security is explicit. evaluate runs the source through new Function, so it executes any embedded expression. That's only safe because the content is the reader's own authenticated vault — never third-party MDX. The seam is documented in code and enforced by convention.

03 Three capabilities, one pipeline

Markdown

Untouched. An MDX note is a superset — tables, code, quotes all render as before. You never give anything up.

Components

A curated registry: Term (definitions fetched live from a glossary note), persistent Checklist, a branching ObjectionSim, course widgets. Unknown tags degrade to text.

Mermaid

A <Mermaid> tag and a pre override that turns a fenced mermaid block into a diagram — other code blocks pass through untouched.

Data & expressions

Read-time {…} expressions and export const datasets mapped into tables, cards, and inline SVG charts. Write the data once.

Mermaid is the only heavy dependency, so it's dynamically imported on first use — its ~0.5 MB lands in its own chunk and never touches the main bundle. Invalid diagram syntax degrades to the raw definition rather than crashing the note.

04 One renderer, two read surfaces

The app has both a Library note route and a Notion-style Pages editor. Both compile MDX through the same renderer — but in Pages it mounts read-only. An mdx note is never loaded into the block editor, which means the save pipeline can't re-serialize and corrupt it. The branch is gated entirely on extension === "mdx"; every ordinary markdown note keeps its exact existing path.

05 Why it's a Parachute story

  • The vault is the substrate; the app is a lens. The same bytes now back static markdown, a block editor, and a runtime component engine — three surfaces, one file.
  • Nothing leaves the note. No build-time MDX, no database, no server round-trip — a static bundle over the vault's REST API. Delete the app and every note still stands as plain text.
  • Edit-once, updates-everywhere. A <Term> pulls its definition from a live glossary note; fix the glossary once and every note that references it updates itself.
  • An escape hatch for the big stuff. Full-page React documents that exceed the note renderer are esbuild-bundled and hosted as their own pages — the same substrate thesis, scaled up to whole apps.
